本文档面向第一次接手这个仓库的人，目标不是只告诉你“这个项目做了什么”，而是让你知道：

1. 请求是怎么在代码里流动的。
2. 每个文件负责什么。
3. 哪些类和函数是主入口。
4. 如果要改功能，应该从哪里下手。

## 1. 先用一句话理解这个仓库

这个仓库实现的是一个**机器人分拣 Agent 中间层**。

它的目标不是直接控制机器人，而是把：

- 输入：`image + instruction`

转换成：

- `object_table.json`
- `plan.json`
- `anygrasp_commands.json`

也就是说，它站在**感知系统**和**机器人执行系统**之间，负责把“模型看懂了什么、任务想做什么”翻译成“执行端能稳定消费的结构化协议”。

---

## 2. 整体执行链路

最重要的主线是下面这条：

```text
image + instruction
  -> segmentation backend
  -> perception
  -> object validation
  -> planning
  -> plan validation
  -> repair
  -> execution adapter
  -> output artifacts
```

如果你想真正看懂这个项目，最好的阅读顺序是：

1. `schemas.py`：先理解数据长什么样。
2. `orchestration.py`：再理解流程怎么走。
3. `backends.py`：看感知后端怎么接。
4. `perception.py`：看目标检测结果怎么转 `ObjectTable`。
5. `planning.py`：看指令怎么转 `Plan`。
6. `execution.py`：看 `Plan` 怎么变成执行命令。
7. `api.py`：看整个系统怎么服务化。

## 3. 目录与文件职责

### 3.1 `src/grasp_agent_middleware/`

这是主代码目录。

## 4. Schema Layer：整个项目的根

### 4.1 为什么先看 `schemas.py`

因为这个项目的核心思想不是“让模型生成一段话”，而是“让所有中间结果都落到强约束结构里”。

换句话说，后面的 perception、planning、execution 都不是围绕 prompt 拼装，而是围绕 schema 在工作。

### 4.2 最关键的几个数据结构

#### `ImageInput`

这个类定义系统如何接收图像输入。

支持五种输入源：

- `image_url`
- `image_path`
- `image_b64`
- `camera_index`
- `stream_url`

并且通过 `model_validator` 强制要求**五选一**。这点很重要，因为它保证了 API 输入不会出现歧义。

#### `BoundingBox`

用来保存目标框，支持：

- `x_min`
- `y_min`
- `x_max`
- `y_max`
- `coordinate_space`

目前默认使用 `normalized` 坐标，也就是 `[0, 1]` 范围。这个设计的好处是 backend 不同、图片尺寸不同，但对下游统一。

#### `MaskRef`

不是直接把 mask 像素塞进对象结构，而是用一个引用来表示 mask。

这样做的好处：

- schema 更轻
- 不强绑定某种 mask 表示方式
- 后续可以替换成 polygon、RLE、PNG、NPY 等格式

#### `ObjectInstance`

这是感知结果里最核心的单个对象。

它里面包括：

- `object_id`
- `label`
- `canonical_label`
- `attributes`
- `bbox`
- `mask`
- `confidence`
- `source`
- `execution_ref`

其中最关键的是 `object_id` 和 `execution_ref`。

因为后面的 planning 和 execution 都是围绕 `object_id` 做引用，而不是围绕“红色杯子”这种自然语言描述。

#### `ObjectTable`

这是 perception 的最终输出。

它相当于一个“当前场景中的可执行对象总表”。

里面有：

- 所有对象 `objects`
- 场景摘要 `scene_summary`
- 不确定对象 `uncertain_objects`
- metadata

后续 planning 并不是重新理解图像，而是只对 `ObjectTable` 做规则匹配和动作生成。

#### `Rule` / `Assignment` / `PlanStep` / `Plan`

这几个类构成 planning 层。

- `Rule`：规则，比如“红色物体去 bin_a”
- `Assignment`：具体对象和目标的绑定关系
- `PlanStep`：单个执行步骤，比如 inspect / pick / place
- `Plan`：完整计划，包括规则、分配、步骤、校验结果、执行契约

#### `ExecutionCommand`

这是执行适配层输出的结构。

它是给 AnyGrasp-style 执行端消费的，不再是高层 planning，而是更贴近机器人执行的命令对象。

#### `PipelineRequest` / `PipelineResponse`

这是 API 层最常用的输入输出契约。

## 5. Config Layer：系统怎么切 backend

`config.py` 负责从环境变量读取配置，然后实例化 `Settings`。

它主要解决两个问题：

1. 让服务在不改代码的情况下切换 backend。
2. 让 demo、测试、真实模型接入使用同一套入口。

主要配置有：

- `LLM_BACKEND`
- `SEGMENTATION_BACKEND`
- `SEGMENTATION_ENDPOINT`
- `ULTRALYTICS_MODEL_PATH`
- `ULTRALYTICS_MODEL_FAMILY`
- `ULTRALYTICS_CONF`
- `ENABLE_COLOR_INFERENCE`
- `OPENAI_COMPAT_BASE_URL`
- `OPENAI_COMPAT_API_KEY`
- `OPENAI_COMPAT_MODEL`

这个文件的特点是简单，但非常关键。因为后面的 `build_workflow()` 其实就是拿 `Settings` 去组装整个系统。

## 6. Backends：模型接入层

`backends.py` 是“外部模型世界”和“本项目内部 schema 世界”之间的适配层。

### 6.1 为什么要单独做 backend 层

因为模型会变：

- 今天用 stub
- 明天用 HTTP service
- 后天换 YOLO
- 再后面可能换 GroundingDINO + SAM

如果 planning 或 API 直接依赖某一个模型的输出格式，整个系统会很快变得不可维护。

所以这里做了一层协议：

- segmentation backend 输出 `list[SegmentationCandidate]`
- LLM backend 输出满足指定 JSON schema 的字典

### 6.2 `StubSegmentationBackend`

这是离线演示和测试时最重要的 backend。

它不看图片内容，而是根据 instruction 中的颜色和形状词做一个确定性的伪感知结果。

它的作用不是追求真实效果，而是：

- 保证整个 pipeline 随时可跑
- 测试 planning / execution / API 是否正常
- 不依赖外部模型和网络

### 6.3 `HttpSegmentationBackend`

这个 backend 的作用是接外部 segmentation 服务。

也就是说，如果你以后有一个单独部署的检测服务，只要它返回：

```json
{"segments": [...]}
```

或者直接返回一个列表，就可以接进来。

### 6.4 `UltralyticsSegmentationBackend`

这是当前真实视觉接入的核心。

它支持：

- YOLO
- SAM
- 图像路径
- URL
- base64
- 摄像头索引
- 视频流 URL

这个 backend 做的事情主要有三步：

1. 把 `ImageInput` 转成 Ultralytics 能接受的 source。
2. 调 `model.predict()`。
3. 把结果转成统一的 `SegmentationCandidate` 列表。

里面还做了几件很实用的事：

- 把 YOLO 的 `xyxy` 转成标准化 bbox
- 从 class name 尝试拆出颜色和形状
- 如果有 mask，则转成 `MaskRef`
- 可选用 OpenCV 做颜色推断

### 6.5 `HeuristicLLMBackend` 和 `OpenAICompatibleLLMBackend`

这两个 backend 对应 semantic extraction。

- `HeuristicLLMBackend`：本地无网络后端，保证系统能离线跑
- `OpenAICompatibleLLMBackend`：接兼容 OpenAI Chat Completions 的服务

这个设计很好的一点是：  
**perception 层不关心语义信息来自启发式还是来自真实 VLM，只关心最后有没有结构化 VisionExtraction。**

## 7. Perception Layer：从检测候选到 ObjectTable

文件：`perception.py`

这个文件最核心的类是 `PerceptionService`。

### 7.1 核心职责

输入：

- `image`
- `instruction`
- `request_id`
- `segments`

输出：

- `ObjectTable`

### 7.2 关键逻辑

#### `canonicalize_label()`

把物体标签做归一化。

例如：

- `block` 归一化成 `cube`
- `sphere` 归一化成 `ball`
- 颜色词会被剔除出 canonical label

这样做的目的，是减少不同检测器或不同 prompt 对同一个物体命名不一致的问题。

#### `complete_attributes()`

如果已有属性不完整，这个函数会从 label 里补颜色和形状。

例如：

- label 是 `red cube`
- attributes.color 为空

那它会自动把颜色补成 `red`。

#### `build_object_table()`

这是 perception 层真正的主函数。

它会：

1. 先尝试调用 `_extract_vision_hints()` 获取更强的语义提示。
2. 遍历每个 `SegmentationCandidate`。
3. 融合 `segment` 和 `vision hint`。
4. 生成稳定的 `object_id`。
5. 构造 `ObjectInstance`。
6. 把所有对象收集到 `ObjectTable` 里。

### 7.3 `execution_ref` 为什么在 perception 阶段就生成

因为 perception 不只是识别物体，它还负责给执行端准备“怎么引用这个物体”。

所以这里会把：

- `frame_id`
- `bbox`
- `mask`
- `object_ref`

一起放进 `ExecutionRef`。

这样 execution 层不需要重新猜“这个对象对应哪块图像区域”，只需要复用 perception 给出的引用。

## 8. Planning Layer：从指令到结构化计划

文件：`planning.py`

核心类：`PlanningService`

这个类是一个**规则驱动的执行就绪规划器**。

### 8.1 `parse_rules()`

这个函数负责把 instruction 解析成规则列表。

它主要做两类事情：

1. 解析颜色 -> 目标位置  
   例如：
   - “red objects to bin A”
   - “blue cylinder into bin B”

2. 识别安全规则  
   如果指令里出现 `fragile`、`glass`、`delicate` 之类的词，就生成保守抓取相关的安全规则。

如果完全匹配不到规则，它会退化成一个默认规则：

- 所有可抓取物体 -> `default_bin`

### 8.2 `generate_assignments()`

这个阶段把“规则”真正绑定到“具体对象”。

也就是从：

- `rule: red -> bin_a`

变成：

- `obj_001 -> bin_a`

这是 planning 里非常关键的一步，因为它把自然语言目标变成了实例级目标。

### 8.3 `generate_steps()`

这个函数把 assignment 转成具体动作步骤。

默认会生成：

- `inspect`（如果对象低置信度）
- `pick`
- `place`

这里的设计思想是：

- planning 不负责真实运动控制
- 但要负责输出执行层需要的动作序列

### 8.4 `validate()`

这是 planning 层的结构校验器。

它会检查：

- assignment 的 object_id 是否真实存在
- step 的 object_id 是否真实存在
- 是否有检测到但没有被分配的对象
- 是否生成了空步骤

### 8.5 `repair()`

repair 的思路是：

如果某些对象没有 assignment，不直接报废整个计划，而是补一个保守兜底策略：

- `inspect`
- `skip`

这样整个系统不会因为某个对象没匹配到规则就完全中断。

这就是它“像 Agent”的地方之一：  
不是只会第一次生成结果，而是会在校验后做修复。

## 9. Execution Layer：把 Plan 变成机器人可消费命令

文件：`execution.py`

核心类：`ExecutionAdapter`

### 9.1 `build_commands()`

它会遍历 `plan.steps`，为每一步生成 `ExecutionCommand`。

这里会把 perception 和 planning 两边的信息接起来：

- 从 `PlanStep` 拿 action / target
- 从 `ObjectTable` 拿 bbox / mask / execution_ref

生成的命令会带：

- `command_id`
- `step_id`
- `action`
- `object_id`
- `target`
- `frame_id`
- `object_ref`
- `bbox`
- `mask`
- `grasp_planner`
- `parameters`

### 9.2 `dry_run()`

这个函数不执行机器人动作，只做结构检查。

它会检查：

- `pick` / `inspect` 有没有 `object_id`
- `object_id` 是否在 `ObjectTable` 中
- `place` 是否有 target

最后输出 `DryRunReport`。

这个设计很重要，因为它让系统在“真正给机器人之前”还有一道机械层面的结构校验。

## 10. Orchestration Layer：系统总编排

文件：`orchestration.py`

这是整个项目的核心文件之一。

### 10.1 `WorkflowState`

它保存一次请求执行过程中的全部状态：

- request
- node_status
- segments
- object_table
- plan
- agent_trace
- execution_commands
- dry_run
- artifacts
- errors

可以把它理解为这次 pipeline 的“运行时上下文”。

### 10.2 `AgentWorkflow`

这是整个系统的总调度器。

你几乎可以把它看成“这个项目真正的后端大脑”。

主要公开方法：

- `run()`：跑完整 pipeline
- `perceive()`：只跑感知
- `plan()`：只跑规划
- `build_execution_commands()`：只生成执行命令
- `dry_run_execution()`：只做 dry-run

### 10.3 `run()` 的完整执行过程

`run()` 做的事情大致是：

1. 生成 `request_id`
2. 初始化 `WorkflowState`
3. 记录 `pipeline_started`
4. 调用 `AgentController.run(state, self._agent_tools())`
5. 保存 `agent_trace.json`
6. 保存 `workflow_state.json`
7. 返回 `PipelineResponse`

如果中途抛异常：

- 记录错误
- dump failure artifact
- 记录 `pipeline_failed`

### 10.4 `_agent_tools()` 的作用

这里定义了一组显式工具节点：

- `segment_tool`
- `perceive_tool`
- `validate_objects_tool`
- `plan_tool`
- `validate_plan_tool`
- `repair_tool`
- `execution_adapter_tool`
- `output_tool`

每个 tool 都有：

- 名字
- 描述
- 是否应该执行 `should_run`
- 真正执行函数 `run`
- 输入摘要函数
- 输出摘要函数

这意味着这个系统不是一个隐式的“大函数调用链”，而是一个可追踪、可回放的 tool workflow。

### 10.5 节点函数

每个 `_xxx_node()` 都只负责一件事：

- `_segmentation_node()`：调用 segmentation backend 并落盘
- `_perception_node()`：构建 `ObjectTable`
- `_perception_validation_node()`：检查对象表是否为空
- `_planning_node()`：生成 `Plan`
- `_plan_validation_node()`：读取 validation 状态
- `_repair_node()`：执行 repair
- `_execution_adapter_node()`：生成命令并 dry-run
- `_output_node()`：写出最终 artifacts

这种设计特别适合后续扩展。

例如以后你想加一个新节点：

- `human_confirm_tool`
- `collision_check_tool`
- `grasp_rank_tool`

只要继续按这个模式新增节点即可。

## 11. Agent Controller：为什么说它有 Agent 感

文件：`agent.py`

这个文件虽然不复杂，但非常关键。

它把每个阶段当成一个 tool 处理，并记录 trace。

为什么这比普通 pipeline 更像 Agent？

因为它具备以下能力：

- 根据状态判断工具要不要执行
- 对每个工具记录决策理由
- 对执行时间和状态做结构化记录
- 让整个运行链可以重放和评估

所以这个项目不是“把几个函数按顺序串起来”，而是“把一组 tool 编排成 workflow”。

---

## 12. Observability：为什么这个项目更像工程项目

文件：`observability.py`

这个文件解决的是“系统出问题时怎么查”的问题。

它主要做三件事：

1. 配置 structured logging
2. 负责写 artifact
3. 负责保存 failure dump

### 12.1 ArtifactStore

通过它，系统会把中间结果写到 `artifacts/{request_id}/` 下。

常见文件包括：

- `segmentation.json`
- `object_table.json`
- `plan.json`
- `anygrasp_commands.json`
- `dry_run.json`
- `agent_trace.json`
- `workflow_state.json`

这对于调试非常重要，因为你可以精确看：

- 是 segmentation 错了
- perception 融合错了
- planning 分配错了
- 还是 execution adapter 输出错了

## 13. FastAPI Service：服务入口怎么工作

文件：`api.py`

### 13.1 `create_app()`

这是 FastAPI 应用工厂。

做的事情是：

1. 从环境变量读取 `Settings`
2. 配置日志
3. 构建 `workflow`
4. 注册所有路由

### 13.2 核心 API

#### `GET /health`

健康检查接口，返回：

- 当前环境
- LLM backend
- segmentation backend
- agent integration

#### `POST /perceive`

只做 perception，输入 image + instruction，输出 `ObjectTable`。

#### `POST /plan`

输入 `ObjectTable + instruction`，输出 `Plan`。

#### `POST /pipeline`

完整端到端接口，最常用。

#### `POST /execution/commands`

把 `ObjectTable + Plan` 转成执行命令。

#### `POST /execution/dry-run`

做 dry-run 检查。

#### `POST /visualize/overlay`

生成 SVG overlay，用来可视化 bbox。

#### `GET /schemas`

返回 schema bundle。

#### `GET /integrations/langchain/tools`

返回 LangChain 工具清单。

### 13.3 Demo 页面

`/demo` 会返回一段 HTML。

它不是复杂前端工程，而是一个轻量调试页面。

主要逻辑：

1. 输入 image 和 instruction
2. 调 `/pipeline`
3. 再调 `/visualize/overlay`
4. 把 ObjectTable、Plan、Trace、Commands 和 Overlay 展示出来

这个页面的价值不在于产品化，而在于调试效率。

## 14. LangChain Adapter：为什么是“可选适配层”

文件：`langchain_adapter.py`

它的设计思路很明确：

- 核心 workflow 继续由本项目自己控制
- LangChain 只是一个适配层，而不是系统底座

这样做的好处是：

1. 保持核心逻辑独立，不被框架绑死
2. 仍然能接 LangChain 工具生态
3. 面试时也能说明：我理解 LangChain，但不会为了框架牺牲系统可控性

它当前暴露了 5 个工具：

- `robotic_sorting_pipeline`
- `robotic_perceive`
- `robotic_plan`
- `robotic_execution_commands`
- `robotic_execution_dry_run`

## 15. CLI、Schema 导出与 Eval

### 15.1 `cli.py`

用于本地调试，不一定非要起服务。

可以直接跑请求文件，打印：

- pipeline 响应
- trace
- commands

### 15.2 `scripts/export_schemas.py`

读取 `json_schema_bundle()`，把当前 schema 导出到 `schemas/` 目录。

这让项目对外的契约非常清晰。

### 15.3 `scripts/run_eval.py`

这个脚本会：

1. 读取 `samples/eval_set.jsonl`
2. 逐条跑完整 pipeline
3. 验证 response schema
4. 统计 assignment、trace、artifact、dry-run 等指标
5. 写出 Markdown / JSON / CSV 报告

这一步很加分，因为它说明项目不是“跑通一条 case”，而是有批量验证能力。

## 16. 一次完整请求是怎么走完的

这里用 `/pipeline` 举例。

### 第一步：API 收到请求

`api.py` 中的 `pipeline()` 接收 `PipelineRequest`，调用：

```python
_workflow(app).run(request)
```

### 第二步：`AgentWorkflow.run()`

它会创建 `WorkflowState`，然后把这个 state 交给 `AgentController`。

### 第三步：分割节点

`segment_tool` 被触发，调用 backend，拿到 `SegmentationCandidate` 列表，并保存 `segmentation.json`。

### 第四步：感知节点

`perceive_tool` 把 `segments` 转成 `ObjectTable`。

这里会补：

- canonical label
- color / shape
- confidence
- execution_ref

### 第五步：对象校验

检查 `ObjectTable` 是否为空，是否有明显错误。

### 第六步：规划节点

根据 instruction 生成：

- rules
- assignments
- steps
- validation

### 第七步：repair

如果 validation 出 warning 或 failed，就走 repair，补兜底步骤。

### 第八步：执行适配

把 `Plan` 转成 `ExecutionCommand`，再做 dry-run。

### 第九步：输出

把 object table、plan、commands、dry_run、trace、workflow_state 等全部写到 artifacts。

### 第十步：API 返回 `PipelineResponse`

所以最终返回的不只是 object_table 和 plan，而是一整套带 trace 和 artifacts 的执行链结果。

## 18. 这个项目最值得讲的工程点

如果你是面试或答辩使用，可以重点讲下面这些：

1. **强约束 schema**  
   不是自由文本，而是 execution-ready contract。

2. **多阶段 workflow**  
   perception -> validation -> planning -> repair -> execution。

3. **可插拔 backend**  
   stub、HTTP、YOLO、SAM、OpenAI-compatible 都能切。

4. **服务化**  
   不是 notebook demo，而是 FastAPI 服务。

5. **可观测性**  
   有 request id、trace、artifacts、failure dump。

6. **可评测**  
   有 eval set，不是只看单例成功。

7. **面向执行端**  
   最终输出可交给 AnyGrasp-style 消费。

## 19. 当前代码的边界和不足

为了避免误解，这里也要说清楚：

1. 当前 planning 主要是规则驱动，不是复杂任务规划器。
2. 默认 perception 仍然有很强的 stub / heuristic 成分。
3. 真正的机器人控制、抓取位姿生成、ROS 对接还没有在这个仓库里完成。
4. Eval 目前更偏 contract validity 和 behavior correctness，还不是视觉精度 benchmark。

这些不是缺点，而是当前系统边界。

## 22. 最后总结

这个仓库最本质的价值是：

**它不是一个“调用模型输出一句话计划”的 demo，而是一个把多模态感知、结构化规划和执行端引用连接起来的 Agent 中间层。**

如果你用一句话向别人解释这个项目，可以说：

> 它把 image + instruction 转换成机器人执行端真正能消费的结构化对象表、执行计划和命令引用，并且整个过程可校验、可追踪、可服务化、可评测。
